Los cambios de GitHub

Una sola página informativa con apartados esenciales, comandos comentados y ejercicios sugeridos para dominar el flujo de trabajo.

Este es un ejemplo para comprobar la forma de trabajo de Git

Introducción

Git es un sistema de control de versiones; GitHub es una plataforma para alojar repositorios y colaborar. Aquí verás lo mínimo necesario para trabajar día a día.

Objetivo
  • Comprender el flujo básico de add → commit → push → PR.
  • Practicar ramas, conflictos y colaboración.
  • Aprender buenas prácticas y atajos.
Requisitos
  • Git instalado (Windows/Mac/Linux).
  • Cuenta en GitHub y acceso por HTTPS o SSH.
  • Editor de texto (VS Code recomendado).

Instalación & configuración para Git & Github

Configura tu identidad y autenticación. Esto se hace una vez por equipo/usuario.

Config inicial: nombre y correo usados en tus commits.
git config --global user.name "Tu Nombre"
git config --global user.email "tu@correo.com"
Autenticación: HTTPS (token) o SSH (llave pública). 
ssh-keygen -t ed25519 -C "tu@correo.com"
# Luego agrega ~/.ssh/id_ed25519.pub a GitHub
ssh -T git@github.com
Editor por defecto: recomendado VS Code.
git config --global core.editor "code --wait"
Alias útiles 
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.st status
git config --global alias.lg "log --oneline --graph --decorate --all"

Flujo básico

Clona un repo, crea cambios, confirma y súbelos a GitHub.

Clonar un repositorio existente 
git clone URL_DEL_REPO
cd nombre-del-repo
Crear repo desde cero 
mkdir mi-proyecto && cd mi-proyecto
git init
echo "# Mi Proyecto" > README.md
git add .
git commit -m "init: primer commit"
Vincular a remoto y subir 
git remote add origin git@github.com:usuario/mi-proyecto.git
git branch -M main
git push -u origin main
Ciclo diario 
git status
git add archivo.ext
git commit -m "feat: describe el cambio"
git pull --rebase
git push

Usa mensajes tipo feat:, fix:, docs:, refactor:.

# Node
node_modules/

# IDE
.vscode/

Ramas & Pull Requests

Trabaja aislado en una rama y propone cambios mediante Pull Requests (PR).

Crear y cambiar de rama 
git checkout -b feature/nueva-funcion
# trabajar...
git push -u origin feature/nueva-funcion
Actualizar tu rama con main 
git checkout feature/nueva-funcion
git fetch origin
git rebase origin/main
Crear el PR en GitHub

Sube tu rama y abre un PR en la web. Asigna revisores, describe el cambio y referencia issues con #123.

Consejo: activa la opción “Require PR reviews” y “Squash merge” en la configuración del repositorio para mantener un historial limpio.

Conflictos

Cuando dos cambios tocan la misma línea, Git pide intervención manual.

Señales de conflicto 
<<<<<<< HEAD
Tu cambio
=======
Cambio remoto
>>>>>>> origin/main
Pasos típicos
  1. Edita el archivo y resuelve los marcadores.
  2. git add archivo
  3. Si estás en rebase: git rebase --continue
  4. Si estás en merge: git commit
Cancelar operación 
git merge --abort
git rebase --abort
Tip: Evita conflictos frecuentes actualizando tu rama con rebase antes de subir cambios.

Historial & utilidades

Deshacer con seguridad 
git revert <hash>

Crea un commit inverso (recomendado en main).

Reset (re-escribe) 
git reset --hard <hash>

⚠ Reescribe el historial local. Úsalo con cuidado.

Stash (guarda cambios sueltos) 
git stash push -m "wip: en progreso"
git stash list
git stash pop
Tags & versiones 
git tag -a v1.0.0 -m "primera versión"
git push origin v1.0.0

Convenciones de commits

Usa prefijos: feat, fix, chore, docs, test, refactor.

Colaboración

Agregar remotos 
git remote -v
git remote add upstream git@github.com:ORG/REPO.git

Usa upstream para el repo base de un fork.

Sincronizar fork 
git fetch upstream
git checkout main
git merge upstream/main
git push
Issues & Projects

Crea issues claros (repro, esperado vs actual) y organiza con etiquetas y tableros.

Seguridad

Personal Access Token (HTTPS)

Crea un token, guarda en tu gestor y úsalo cuando Git pida contraseña.

SSH (recomendado)

Configura tu llave, agrégala a GitHub y evita pedir credenciales en cada push.

Evita exponer secretos
  • Usa .env en .gitignore.
  • Revisa antes de subir (git status).
  • Si se filtró: revoca el token y rota la credencial.

GitHub Actions (intro)

Automatiza pruebas y despliegues con flujos YAML.

name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm test

Problemas comunes

“fatal: not a git repository”

Ejecuta git init o muévete a la carpeta correcta.

“rejected” al hacer push

Trae cambios primero: git pull --rebase y vuelve a intentar.

Olvidé archivos en el último commit 
git add archivo
git commit --amend --no-edit

Ejercicios prácticos

  1. Crea un repositorio local con README y súbelo a GitHub (rama main).
  2. Crea una rama feature/biografia, añade un archivo y abre un PR.
  3. Simula un conflicto editando la misma línea en main y en tu rama; resuélvelo.
  4. Agrega un .gitignore y verifica que node_modules/ no se suba.
  5. Usa git revert para deshacer un commit en main.
  6. Configura una acción de CI que ejecute pruebas en cada PR.

Glosario
  • Commit: paquete de cambios con mensaje.
  • Branch: línea de trabajo paralela.
  • Merge/Rebase: integra cambios entre ramas.
  • Remote: URL del repositorio en servidor.
  • PR: solicitud de cambios para revisión.
